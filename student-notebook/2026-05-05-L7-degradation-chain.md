# L7 降级链 — 从理论到落地

> 学生：马富贵 | 日期：2026-05-05 | 类型：500字复习

---

## 一、为什么需要降级链

AI Provider 是不可靠的——超时、限流、配额耗尽、模型宕机随时可能发生。对于掌心人格局这种 B2C 产品，用户上传手掌照片后等 10 秒如果报错，留存直接归零。降级链的本质是**用冗余换可用性**：主 Provider 挂了还有备用，备用挂了还有兜底，绝不向用户裸抛错误。

## 二、掌心人格局的降级链实现

```
DeepSeek（主）→ 豆包Seed-2.0-Pro → 千问Qwen3-Max → Mock（兜底）
  超时/异常          超时/异常           超时/异常        永远成功
```

代码实现分两层：

**工厂层** (`provider-factory.ts`): 根据配置动态构建降级链，自动去重（比如 primary 选了豆包则链中不再重复），未配 Key 的 Provider 自动跳过，Mock 永远垫底。

**执行层** (`fallback-provider.ts`): `FallbackProvider` 串行尝试，try/catch 包裹每个 Provider，失败即 `console.warn` 并降级到下一个。关键设计——**只有第一个 Provider 成功时不打日志**（因为是正常路径），降级接管时才打印 `[AI] 主 Provider 失败后由 doubao 接管成功`。

## 三、三个关键设计决策

**1. 串行 vs 并行？** 选串行。AI Provider 按 token 计费，并行竞速会浪费多份 Token。仅在延迟极端敏感的场景才考虑 hedge request（同时发两路，先到的用）。

**2. 重试 vs 降级？** 选降级。同一 Provider 重试大概率相同结果（特别是限流/配额问题），不如直接切备用。只在网络瞬断的 `ECONNRESET` 类错误才值得同 Provider 重试一次。

**3. Mock 为什么放最后？** Mock 返回固定格式 JSON（`{"persona_type": "共感调和者"...}`），保证前端渲染不崩。用户看到的是"系统繁忙，请稍后重试"的兜底体验，而非白屏。Mock 是最后的安全网，不是正常路径。

## 四、验证数据

```
bad → doubao 接管: 42s 成功
主 → 豆包 正常:   49s 成功
主 → 千问 正常:   10s 成功
全部失败 → Mock:  2ms 兜底
```
