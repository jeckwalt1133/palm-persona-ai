# 批判题模拟练习

> 考试范围：MCP 协议 + Agent 安全 + 多Agent + Skill/Hook 综合分析

---

## 题目：分析 MCP 协议在当前 Agent 生态中的定位、局限性和改进方向

### 论点 1：MCP 的生态定位

MCP 试图成为 AI Agent 的"USB-C 接口"——一个标准化的数据/工具接入协议。类比：

```
没有 MCP 的世界:
┌──────┐     ┌──────────┐
│ Agent│────→│ 插件A     │ ← 自定义集成
│      │────→│ REST API  │ ← HTTP 调用
│      │────→│ 数据库直连 │ ← SQL 查询
│      │────→│ 内部工具   │ ← 私有 RPC
└──────┘     └──────────┘

有 MCP 的世界:
┌──────┐     ┌──────────┐
│ Agent│────→│ MCP Server│ ← 统一 JSON-RPC
│      │     │  ├─ 数据源A│
│      │     │  ├─ 工具B  │
│      │     │  └─ 模板C  │
└──────┘     └──────────┘
```

**核心价值**：标准化替代碎片化。一次开发 MCP Server，所有 MCP 兼容的 Client 都能用。

### 论点 2：局限性分析（有数据支撑）

| 问题 | 严重度 | 分析 | 数据/证据 |
|------|--------|------|-----------|
| **JSON-RPC 耦合** | 高 | 非 JS 生态不自。Python/Go/Rust SDK 社区维护，质量参差 | Python SDK v0.4 仍缺部分功能 |
| **STDIO 的分布式缺陷** | 中 | STDIO 绑定子进程生命周期，无法用于远程/分布式部署 | Streamable HTTP 2025才加入 |
| **无生产级安全** | 高 | 无 SSO/RBAC/审计日志 | 企业部署需自行实现 |
| **无服务端推送标准** | 中 | notifications 机制有限，不适合实时场景 | 需轮询替代 |
| **厂商锁定风险** | 中 | Anthropic 主导，虽有开源协议但决策权在 Anthropic | 类比 React vs 其他框架 |
| **协议版本演进** | 低 | 2024→2025→2026 快速迭代，但向后兼容不错 | 实测 2025-11-25 OK |

### 论点 3：与替代方案的对比

| 方案 | 标准化程度 | 生态 | 成熟度 | 最适合 |
|------|-----------|------|--------|--------|
| **MCP** | 高(官方协议) | 增长中 | 早期 | 通用 AI 数据接入 |
| **OpenAI Function Calling** | 中(OpenAI 绑) | 成熟 | 成熟 | OpenAI 生态 |
| **REST API + OpenAPI** | 高(OpenAPI 标准) | 极成熟 | 极成熟 | 传统 Web 服务 |
| **gRPC** | 高 | 成熟 | 成熟 | 高性能微服务 |
| **直接 Tool 定义** | 低(每个 AI 平台独立) | 碎片化 | 成熟 | 单一平台 |

### 论点 4：对掌心人格局的具体影响

**当前选型是正确的**，因为：
1. 掌心人格局是内部工具（非公开 API）——STDIO 传输足够了
2. 数据桥接只涉及 2 个源（合规词库 + 报告数据）——协议"重量"合适
3. MCP 的原语划分（Tools/Resources/Prompts）匹配项目需求

**但可以改进的**：
1. Tool 定义的 Input Schema 没有版本管理——建议加 `schemaVersion` 字段
2. Resources 不支持分页——report 多了以后需要 cursor-based pagination
3. 没有工具调用审计——建议每次 `tools/call` 记录请求方 + 时间戳
4. 错误码不够细——当前只有 InvalidParams / InvalidRequest / MethodNotFound，缺业务错误码

### 结论

MCP 是**方向正确但仍在进化**的协议。掌心人格局当前使用是"够用且正确"的。生产级部署前需要补：鉴权 + 审计 + 分页 + 业务错误码。

## 附加：若题目改为"掌心人格局的技术栈综合批判"

**做得好的：**
1. 分层清晰：Frontend → Server → AI Provider → Shared Packages
2. 安全基线到位：限流、敏感词过滤、原图即删
3. 模块化：Shared 包共享类型/工具/安全
4. 合规优先：27 项禁用词 + 替代表达 + 免责

**有风险的：**
1. **共享包 import 链路脆弱**：`../../../packages/shared-safety/src/index.js` 是相对路径，重构时容易断
2. **无多环境配置**：dev/test/prod 用同一套配置
3. **测试覆盖偏窄**：有单元测试但缺少集成测试和 E2E 测试
4. **MCP Server 是单点**：目前只连 2 个数据源，扩展后可能需要拆分 Server

**改进建议（按优先级）：**
1. 高：加 E2E 测试（至少覆盖标准黄金路径）
2. 高：MCP Server 加调用日志（调试 + 审计）
3. 中：Shared 包用 workspace protocol 而非相对路径
4. 中：环境区分（dev/prod .env）
5. 低：MCP 升级 Streamable HTTP 支持远程部署
