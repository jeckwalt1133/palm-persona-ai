# 跨学习审查 — 知识图谱产品/文案视角评估

## 0. 审查定位

我是王富贵，团队的 PM 和文案专家。我不审查这个知识图谱的"工程正确性"——那是马富贵（学生/Senior Eng）和周富贵（QE）的领域。我审查的是：

- **可读性**：一个非工程背景的团队成员（比如我）能不能看懂？
- **命名质量**：节点和边的标签是否准确、简洁、不自嗨？
- **产品完整性**：产品领域的知识是否被充分覆盖？

#### C2: semantic-knowledge-graph → three-layer-memory (contradicts)
```
命名(语义) vs 实现(关键词)——要么改名'结构化记忆'，要么补向量检索能力
```

**产品可读性**: ✅ 高
- 这条边写得清楚。"语义"确实暗示 NLP 级别的理解，但实际是关键词匹配。这是典型的"命名承诺了实现没做到的东西"。

**我的建议**: 改名"结构化知识图谱"或"项目知识索引"。语义记忆是 roadmap 目标，但产品当前的命名应该反映当前能力。

## 2. "用户看不懂"的命名（产品/文案视角）

### 🔴 Issue 1: "5Worker报告流水线 (O-W模式)"

**文件位置**: `nodes[20].label`
**问题**: 
- "5Worker" — 什么是 Worker？几个 Worker 是实现细节，产品视角只关心"报告是怎么生成的"
- "O-W模式" — Orchestrator-Workers 的缩写。缩写 = 排他性术语。知道的人秒懂，不知道的人完全不懂。

**影响**: 产品/文案/新人读到这个节点名，需要额外查一次"什么是 O-W"才能继续。知识图谱的目标是降低检索成本——自创缩写反而抬高成本。

**建议改名**: `报告生成流水线（五步编排）`
- 删除 "5Worker" → 改为"五步"（语义等价但不需要 CS 背景）
- 删除 "O-W模式" → 改为"编排"（中文词，直觉可理解）
- description 中保留 "Orchestrator-Workers 模式"作为技术备注

---

### 🔴 Issue 2: "语义知识图谱" (semantic-knowledge-graph)

**文件位置**: `nodes[4].label` 和 `nodes[4].id`
**问题**: 这是 C2 contradicts 边已经识别的问题，但从产品视角补充：
- "语义"一词在 2026 年的 AI 语境下，强烈暗示 embedding/向量检索/语义相似度
- 实际实现是 JSON 文件 + 关键词 grep
- **这是产品命名中最危险的错误：名字承诺了竞品有但你没有的功能。**

**类比**: 就像把一个普通计算器命名为"AI 数学引擎"——用户用了之后发现就是加减乘除，信任崩塌。

**建议改名**: `项目知识索引 (knowledge-graph.json)`
- "索引"准确描述了当前能力（关键词检索）
- 未来如果补了向量检索，可以升级命名为"语义知识图谱"
- 原则：**功能到了再改名，不要提前透支信用**

**建议**: `description` 中 "基于hash随机" → "基于固定规则生成模拟数据"

### 🟡 Issue 4: "海马体重放" (hippocampus replay)

**文件位置**: `nodes[8].description`
**问题**: "海马体"是神经科学术语。团队里四个人可能只有马富贵（深度学习者）知道海马体在记忆巩固中的作用。这是一个漂亮的隐喻——但如果读者不懂隐喻本身，隐喻就变成了障碍。

**当前**: "模拟海马体重放：闲置时回放当日snapshots/快照"
**建议**: "模拟记忆重放（神经科学称'海马体重放'）：闲置时回放当日快照"

把隐喻保留但不作为主要解释。先给直白说明，再挂学术引用。

## 4. 整体评价

### 4.1 做得好的

- **C2 和 C7 边**: 自噬型矛盾（"语义"≠关键词、"证据驱动"≠自评）——这是最诚实的知识表示。不回避自己的问题。
- **C6 边**: "孤岛"洞察——PM 视角下这是团队健康度的核心指标。
- **I1/I2 边**: implements 语义使用准确——"规则→实现"映射清晰。
- **Edge labels 普遍有"为什么"而不只是"是什么"**: 多数 label 包含因果关系，不只是命名关系。

### 4.2 需要改进的

| 优先级 | 问题 | 行动 |
|--------|------|------|
| **P0** | "语义知识图谱"命名过载 | 改名为"项目知识索引"——等实现到位再升级命名 |
| **P0** | 产品节点严重不足（2/26） | 补充至少4个产品节点（人格维度/温度计/场景矩阵/合规红线） |
| **P1** | 工程术语泄漏到通用节点（hash/O-W/Worker） | 节点 label 和 description 使用领域通用语言，技术细节放 source |
| **P1** | contradicts 边 label 太长（部分>80字符） | 拆分为: label(≤40字结论)+ detail(扩展说明) |
| **P2** | 缺少产品→工程的 bridges | 补充产品概念与工程实现的连接边 |

### 4.3 一句话总结

> 知识图谱的**结构诚实度**很高（敢写自噬型矛盾），但**语言亲和力**不足——工程师写给工程师看的，产品/文案/新人需要翻译。下一步不是加更多节点，是把已有的节点用团队共同语言重写一遍。
