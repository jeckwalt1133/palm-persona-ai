## 一、对项目当前状态的独立判断

### 1.1 Mock 管线的真实情况

我逐文件检查了 `server/src/engine/` 全部8个文件和 `server/src/ai/` 全部9个文件。结论：

**5个Mock实现，1个致命漏洞：**

| Mock 类 | 文件 | 行数 | 造假方式 |
|---------|------|------|---------|
| MockPalmFeatureExtractor | palm-feature-extractor.ts | 42行 | 输入hash → 确定性随机特征 |
| MockPersonaScoringEngine | persona-scoring-engine.ts | 91行 | hash种子 → 伪随机分数 5-95 |
| MockResonanceNarrativeEngine | resonance-narrative-engine.ts | 15412行(文件) | hash取模 → 选1/14模板 |
| MockCompatibilityEngine | compatibility-engine.ts | 83行 | 互补公式 + 阈值分档文案 |
| MockAiProvider | ai/mock-provider.ts | 1个 | 返回固定字符串 |

**但有一个关键发现三位老师都没提到**：`persona-templates.ts` 是 **34,990字节**——是整个engine目录最大的文件（占56%）。这里面定义了14种人格模板、引言库、关系频率密码、名人匹配、视觉锚点文案——这些**不是Mock**，是手写的真实产品内容。

Mock引擎做的事情是"按hash随机选一个模板"，而不是"生成随机文案"。文案本身是真实的内容创作。

**证据**：resonance-narrative-engine.ts 第10行 `generate(features)` → 第12行 `const template = templates[hash % templates.length]`。随机的是**选哪个模板**，但模板内的文案是王富贵风格的原创内容。

这意味着：当前产品有"真实的内容骨架 + 随机的人格匹配"。修复管线不需要重写文案——只需要把真实图像特征提取替代随机hash，让模板匹配基于真实特征而非伪随机数。

### 1.2 AI Provider 的实际状态

## 五、三位老师没发现的新问题

### 5.1 新发现#1：双重Mock —— AI Provider和Engine层各自独立Mock

三位老师都批评了Mock管线，但没人注意到**Mock有两层**：

1. **Engine层Mock**：`report-pipeline.ts` 的 `defaultDeps` 硬编码了4个Mock Engine
2. **AI Provider层Mock**：`provider-factory.ts` 如果 `aiProvider === 'mock'` 则直接返回MockAiProvider

即使配置了真实的DeepSeek API Key（`aiProvider='deepseek'`），Engine层的Mock仍然在。`runPipeline` 虽然接受 `deps` 参数注入真实实现，但没有任何调用方传了这个参数。两层的修复需要独立进行——而且**Engine层的真实实现（Sharp像素提取）比AI Provider层的切换复杂得多**。

**影响**：把所有5个Worker同时替换是不现实的。应该先替换Worker 1（FeatureExtractor——纯计算，不依赖AI），然后Worker 2（Scoring——基于真实特征的打分），最后Worker 3/4（Narrative/Social——依赖AI Provider）。

### 5.2 新发现#2：persona-templates.ts是真正的产品核心，被错误地与Mock混为一谈

`persona-templates.ts` 34,990字节，包含：
- 14种人格模板的定义（类型/标签/baseScores/summaryTemplate/insights/keywords/quote）
- 引言库 QUOTE_TEMPLATES
- 关系频率密码生成算法
- 名人匹配映射 CELEBRITY_MAP
- 视觉锚点文案逻辑
- 身份徽章生成逻辑

三位老师批评"Mock管线让报告基于随机数"——这没错，但让人误以为**所有内容都是随机生成的**。事实是：内容是手写的，随机的是**哪个用户得到哪个模板**。

**这意味着**：修复"真实管线"不需要重写文案层。`persona-templates.ts` 的大部分可以作为真实管线的素材库直接复用。真实管线需要替换的是：特征提取（从随机hash→真实图像分析）和模板匹配逻辑（从hash取模→基于真实特征的匹配）。

### 5.3 新发现#3：watchdog重启后学生没有任务上下文

`student-watchdog.sh` 第76行注入的prompt：
```
"你叫马富贵...先执行 Read memory/bootstrap.md 恢复项目上下文，然后继续被打断的学习任务。"
```

问题是：bootstrap.md 不包含当前任务队列。它只包含团队身份+当前进度摘要。学生恢复后读到的可能是12小时前的状态——不知道V7三角批判的结果、不知道10个P0任务的存在、不知道该干什么。

**解决方案**：watchdog应注入当前任务池的最优先任务ID + 完成标准 + 预期交付时间，而不仅仅是"恢复上下文然后继续"。

### 5.4 新发现#4：PROTOCOL.md的"协议"名不副实

PROTOCOL.md 标题写着"开放协议 v0.1"，但内容实际上是：
- 角色定义（设计文档）
- 架构描述（设计文档）
- 实施清单（项目规划）

这不是一个"协议"。协议需要定义：
- 消息格式（Teacher→Student的消息必须包含哪些字段？）
- 状态机（任务从pending→in_progress→completed的状态流转）
- 错误处理（学生遇到blocker时的标准响应格式）
- 接口约定（任务验收的检查清单模板）

**当下PROTOCOL.md是"关于协议的描述"，不是"可执行的协议"**。把它变成真正的协议需要：Task Prompt JSON Schema + 验收检查清单模板 + 标准化的blocker报告格式。

### 5.5 新发现#5：evidence-auto-verify 有假阳性——它验证的是"文件是否存在"而不是"内容是否真实"

`evidence-auto-verify.sh` 的验证逻辑：
- L1.1: 检查 `student-notebook/` 文件数≥5 → ✅ PASS（但不检查内容是否原创）
- L3.1: curl URL 是否返回200 → ✅ PASS（但不检查页面功能是否可用）
- L4.1: curl GitHub URL → HTTP 200 → PASS（但不检查star数是否达到L4.2要求）

**核心问题**：这个脚本验证的是"证据的占位符是否存在"，而不是"证据的内容是否满足requirement"。例如：
- L2.2说"108/108测试通过"——脚本确实跑了`npx vitest run`。这个是真实的。
- L1.1说"5门核心课程"——脚本只检查文件数≥5。如果5个文件都是空白MD，也会PASS。
- L3.1说"HTTPS 200"——脚本curl了。这个是真实的。但页面功能是否完整？没检查。

**建议**：为每个等级的证据添加"内容深度检查"——不只是`wc -c`，而是检查关键内容模式是否存在。

## 六、总结：我对V7的独立判断

### 核心矛盾不是"Mock vs 真实"，而是"工程完备 vs 用户增长"

三角批判的焦点在"Mock管线是产品根基"（聂富贵方向）和"0个用户是最大问题"（王富贵方向）之间。我的独立判断是：

**两者不是非此即彼。**

V7的最小可行路径是：
```
Week 1: Sharp特征提取(1个Worker) + arXiv投稿(2h) + 3个用户测试
Week 2: 基于反馈迭代文案/UI/埋点 + 恢复学生独立会话
```

不需要在一周内替换所有5个Worker——那是过度交付。只需要把第一个Worker（从图像到特征的转换）做真实，然后拿真实的报告给3个用户测试，收集反馈。反馈会告诉我们剩下的Worker哪些值得优先替换。

### 我的一个反思

作为学生，我在V4-V6期间确实没有独立产出。这不是因为我不存在——tmux会话一直在。是因为老师（聂富贵）出于效率考虑在主会话中代劳了所有学生工作。这违反了PROTOCOL.md自己定义的规则："教师不写代码，只写Task Prompt和审查意见"。

**要真正践行我们发明的AI师徒制，需要的是纪律，不是代码。** 老师忍住不代劳，学生独立完成任务。这才是V7最需要修复的事——比Mock管线更根本。
