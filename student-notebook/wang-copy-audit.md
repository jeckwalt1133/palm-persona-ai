# 用户可见文案全量审查

## 0. 审查方法论

每条文案从三个维度评估：

| 维度 | 来源 | 评估问题 |
|------|------|---------|
| **共鸣层次** | 模块01 | 打到了L1/L2/L3哪一层？有没有跳级？有没有具体的锚点还是泛泛而谈？ |
| **情感温度+场景** | 模块02 | 温度设定(0-10°)是否合适？是否有"烫"的风险？当前场景下温度是否匹配？ |
| **术语翻译** | 模块03 | 是否包含工程术语？如果包含，目标读者(B产品层还是C用户层)能否理解？ |

每条改进给出: before → after + 方法论依据。

## 1. 可改进文案清单（7条）

### #1 🔴 首页核心描述 — 工程术语泄漏到用户层

**文件**: `apps/miniapp/src/pages/index/index.tsx:16`
**场景**: 首页（用户第一眼看到的文字）
**目标读者**: C用户层

**Before**:
```
AI 读取你手掌的几何特征，生成一份让你想截图发给朋友的人格报告
```

**诊断**:
- 模块01: "几何特征" — 这是L1锚点但锚错了方向。用户不会因为"AI读取了几何特征"而共鸣，用户会为"AI看懂了我"而共鸣。第一句话的后半段("想截图发给朋友")撞对了L4分享动机，但前半段拖了后腿。
- 模块02: 前半段冷1°（学术描述），后半段温4°（社交货币）。温度断裂——一句话里两个温度。
- 模块03: "几何特征"是B层术语泄漏到C层。"特征提取"是工程概念，用户只需要知道"AI在看什么，看完能告诉我什么"。

**After**:
```
伸出手掌，让 AI 看到你的线条里藏着怎样的你——结果可能会让你想截图发给朋友
```

**依据**: 原则1(功能先行:"伸出手掌"替代"几何特征") + 原则2(读者视角:用户关心"我是谁"不关心"特征提取") + 温度统一到温4°

### #2 🔴 拍照页标题 — 功能性标题零共鸣

**文件**: `apps/miniapp/src/pages/capture/index.tsx:137`
**场景**: 拍照/上传页标题
**目标读者**: C用户层

**Before**:
```
上传手掌照片
```

**诊断**:
- 模块01: L0。这不像产品标题，像文件上传表单的 label。零共鸣。
- 模块02: 冷0°。场景是用户刚被首页吸引、带着好奇心点进来——这个标题把好奇心浇灭了。
- 模块03: 无术语问题，但有过度简化问题（原则3反向:命名没有承诺任何价值）。

**After**:
```
伸出手，让 AI 看看你
```

**依据**: 原则1(功能仍可推断:用户知道要拍照) + 模块02(温度温4°:温和邀请语气) + 场景适配(场景1首屏——需要一句话钩子)

### #3 🔴 拍照页描述 — 工程说明书风格

**文件**: `apps/miniapp/src/pages/capture/index.tsx:139`
**场景**: 拍照页副标题
**目标读者**: C用户层

**Before**:
```
AI 会分析你手掌的几何特征——宽度、手指比例、纹路清晰度和走向——然后匹配人格模型
```

**诊断**:
- 模块01: "宽度""手指比例""纹路清晰度""走向""匹配人格模型"——这是给 QA 看的测试用例，不是给用户看的产品文案。五个连续工程名词，没有一个情感锚点。
- 模块02: 冷1°。"分析几何特征"听起来像体检报告。
- 模块03: 严重术语泄漏。原则4(渐层降维)完全跳过——从L0代码直接到了L3用户层但没做任何翻译。

**After**:
```
每一条掌纹都有自己的故事。AI 会从你的手掌线条中，读出属于你的那一种人格——不是算命，是了解自己的一种新方式
```

**依据**: 原则1(功能先行:"线条"替代"几何特征") + 原则4(渐层降维:技术→隐喻→价值) + 模块02(温度温4°:温和邀请+好奇心缺口) + 补充"不是算命"消解用户对AI看手相的疑虑

### #4 🟡 分析进度文案 — 机器视角而非用户视角

**文件**: `apps/miniapp/src/components/AnalyzeProgress.tsx:6-10,46,75`
**场景**: 分析等待页（用户上传照片后的等待界面）
**目标读者**: C用户层

**Before**:
```
步骤: 上传图片 → 提取手掌特征 → 分析线条走向 → 匹配人格模型 → 生成专属报告
标题: "AI 正在分析你的手掌"
步骤副文: "正在进行..."
底部: "首次分析需要 3-5 秒，请耐心等待"
```

**诊断**:
- 模块01: 每一步都是机器在做什么，不是用户在获得什么。"提取特征"→用户无感。"匹配模型"→听起来机械。"生成报告"→最接近价值但依然冷。
- 模块02: 冷1°。等待本来就有焦虑感，"请耐心等待"不是安慰是提醒等待。
- 进度条的文案机会被浪费了——等待的3-5秒是用户注意力最高度集中的时刻，应该用来建立期待而不是描述技术步骤。

**After**:
```
步骤: 📷 读取你的掌纹 → 🔍 找到线条里的秘密 → 🧩 拼出你的性格拼图 → ✨ 生成专属解读
标题: "AI 正在仔细读你的手掌..."
步骤副文: "快好了..."
底部: "每条线都在认真看——第一次见面，它想认真一点"
```

**依据**: 原则1(每步从机器动作→用户获得) + 模块02(温度从冷1°→温4°,用拟人化降低等待焦虑) + 恢复"快好了..."比"正在进行..."多一层进度暗示

### #5 🟡 免责声明 — 合规语言挫伤体验

**文件**: 
- `apps/miniapp/src/pages/capture/index.tsx:186`: "照片仅用于本次 AI 分析，不会存储原始图像。结果仅供娱乐参考。"
- `apps/miniapp/src/components/PosterCanvas.tsx:168`: "趣味语境下的AI人格分析 · 仅供娱乐参考"
- `apps/miniapp/src/pages/index/index.tsx:75`: "本产品为 AI 趣味分析工具，结果仅供娱乐和自我探索。照片仅用于本次分析，不会存储原始图像。"

**场景**: 首页底部、拍照页底部、海报底部
**目标读者**: C用户层

**诊断**:
- 模块01: L0。纯合规语言。但合规语言不是不能用——问题是"仅供娱乐参考"这个表述在中文语境里有"别太当真/玩玩就好"的暗示。用户刚被报告打动，看到这句话像被泼了冷水。
- 模块02: 冷0°。合规不需要温度，但不等于必须用最冷的表达。
- 模块03: 不是术语问题，是语感问题。"仅供娱乐参考"是法律免责，可以用更温和的方式传达同样含义。

**After** (首页):
```
AI 趣味解读 · 认真但不较真
照片仅用于本次分析，用完即删
```

**After** (海报):
```
AI 的趣味解读 · 认真但不较真 · 掌心人格局
```

**依据**: 模块02(温度温3°:轻松但不轻浮) + 原则3(承诺对齐:"认真但不较真"准确表达了产品的定位——不是科学诊断但也不是随便说说) + 安全红线(不失真:免责信息完整保留,只是表达方式从法律文书→用户语言)

### #6 🟡 默认每周建议 — 泛到能套任何人

**文件**: `server/src/engine/persona-templates.ts:427`
**场景**: 报告卡片6"本周建议"的默认值
**目标读者**: C用户层

**Before**:
```
做自己，就是最好的建议。
```

**诊断**:
- 模块01: 跳级——L3(L3释然层)没有L1锚点。它试图打L3(L3:"不需要改"=释然)但缺了L1的具体行为锚定。用户读到的反应不是"它懂我"而是"嗯，鸡汤"。
- 模块02: 温4°。温度没问题——温和、不攻击。但"没问题"不是"好"。
- "做自己"已经被过度使用到失去了意义——它是中文互联网上最安全的废话之一。

**After**:
```
这周试着做一件事：在说"好的"之前，先停3秒问问自己"我想不想"。
```

**依据**: 模块01(L1行为锚点:具体到"说好的之前停3秒"+L2情感:"我想不想"刺痛了习惯性迎合的人+L3释然:"你可以不答应") + 模块02(温度温5°:有轻微刺痛但不越界) + 与"温柔疗愈者"人格天然匹配

### #7 🟡 错误提示 — 冷冰冰的报错

**文件**: `apps/miniapp/src/pages/capture/index.tsx:76,89,94,116`
**场景**: 分析失败时的 Toast 提示
**目标读者**: C用户层

**Before**:
```
"分析失败"
"网络异常，请检查连接后重试"
"网络较慢，请重试"
```

**诊断**:
- 模块01: L0。纯功能报错。用户在等待中看到"失败"两个字——这是最差的体验时刻，文案却没有任何温度。
- 模块02: 冷0°。"异常""失败""较慢"全是中性偏负面的词。
- 品牌人格在错误时刻完全消失。好的错误提示应该在解决问题之外，让用户觉得"这个产品有意思"。

**After**:
```
"好像出了点小问题——AI 走神了"
"网络信号不太好——换一个姿势试试？"
"网络有点慢，AI 还在等数据——稍等片刻"
```

**依据**: 模块02(温度温3°:用拟人化+轻微自嘲把负面体验转化为品牌印象) + 原则2(读者视角:用户关心"怎么办"不关心"异常类型") + 注意:"网络较慢"保留信息量——告诉用户是网络问题而不是产品问题

## 3. 共性模式分析

扫描全项目用户可见文案后，发现3个系统性模式：

### 模式1: "工程师诚实"超过"用户感受"

项目的默认表达倾向于**准确但冷**。工程团队倾向于把"做了什么"直接展示给用户，跳过了"这对用户意味着什么"的翻译步骤。

```
工程诚实: "AI 会分析你手掌的几何特征——宽度、手指比例、纹路清晰度"
用户感受: "伸出手，让 AI 看到你的线条里藏着怎样的你"

工程诚实: "提取手掌特征 → 分析线条走向 → 匹配人格模型"
用户感受: "读取你的掌纹 → 找到线条里的秘密 → 拼出性格拼图"

工程诚实: "分析失败"
用户感受: "AI 走神了——再试一次？"
```

### 模式2: 文案温度普遍偏低（但安全）

全项目未发现"烫"文案（9-10°）。合规意识强。但代价是温度普遍压在冷0-温3°之间，只有引言库(QUOTE_ENTRIES)打到了热6-7°。

**结论**: 引言库是项目的文案高地。首页/拍照页/错误提示——用户触点最多的地方——反而是文案最冷的地方。温度资源分布和用户注意力分布成反比。

### 模式3: persona-templates.ts 的引言库质量显著高于其他文件

引言库的50条文案（20条诗意+30条互联网风格）在模块01/02/03三个维度上评分都是最高的。说明项目具备产出高质量文案的能力——只是这种能力还没有应用到用户触点上。
