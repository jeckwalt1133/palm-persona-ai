## 模块2: 王富贵 — 文案共鸣基础 (wang-copywriting-01)

### 核心收获

**三层共鸣模型**：
1. 认知共鸣 — "它说得对" (准确描述)
2. 情感共鸣 — "它理解我" (说出没说出口的感受)
3. 社交共鸣 — "我想让ta看" (帮我对别人表达)

**5人格×分享动机矩阵**是产品文案的底层操作系统。每个文案决定必须经过"这个人格类型的用户分享动机是什么"的思考。

### 关键洞察

"正确但不共鸣" vs "有共鸣"的对比是整个模块的精华：
- "这是一份基于AI的人格分析报告" → "天哪，这说的不就是我吗？"
- 差异不在信息量，在**视角转换**：前者是产品视角(我们做了什么)，后者是用户视角(这对我意味着什么)

### 应用

掌心人格局的所有报告卡片文案都应该经过这个矩阵的审查。当前6张卡片可能主要覆盖了认知共鸣层，情感共鸣和社交共鸣层需要加强。

## 模块3: 王富贵 — 情感分层与场景化文案 (wang-copywriting-02)

### 核心收获

**四级情感深度模型** (与学习标准L1/L2/L3不同，这是文案领域的独立框架):
- L1 认知层: 准确描述→用户点头"说得对"
- L2 情感层: 说出感受→用户卸防"它懂我"
- L3 价值层: 给解释→用户释然"原来这不是我的错"
- L4 行动层: 帮表达→用户分享"我要发给XX看"

**层级不可跳跃**: L1松了，后面的全是空中楼阁。跳过L1直接打L2情感会导致用户反应"呃……是吗？"——因为没有具体行为锚定。

### 六大场景矩阵

| 场景 | 信任半径 | 核心策略 |
|------|---------|---------|
| 朋友圈 | 中 | 一句话钩子+好奇心缺口 |
| 私聊 | 高 | 情感打点+关系锚定 |
| 群聊 | 低 | 话题钩子+低门槛参与 |
| 收藏 | 无 | 深度共鸣+反思空间 |
| 截屏 | 低 | 强力钩子+品牌水印 |
| 二次传播 | 极低 | 极度简练+视觉锚点 |

**文案的质量 = 内容共鸣 × 场景适配度。一个乘数为零，结果为零。**

### 情感温度计 (最实用的工具)

```
冷 0-2° 客观陈述 → 准确但没人分享
温 3-5° 温和共鸣 → 安全区，默认档位
热 6-8° 强烈共情 → 需要2倍L1证据支撑
烫 9-10° 过度煽情 → 禁止使用
```

**关键规则**:
- 默认用温(3-5°)
- 每升1°需要2倍L1证据
- 6°以上必须第二人审查
- 禁止烫(9-10°)

### 与安全模块的交叉

温度计和安全红线的对应关系精妙：
- "不说命运" = 温度不超过7° + 不用L3的"重新定义"模式定义用户命运
- "不说诊断" = 温度不过6° + 不用L3的"溯源归因"做临床判断
- "不说绝对" = 所有温度所有层级的通用约束
- "不说恐吓" = 禁止用L3"预见化"模式做负面预测

**安全红线本质上是对情感温度的硬性约束。**

## 模块4: 周富贵 — 安全左移与CI集成 (zhou-security-02)

### 核心收获

**三个检测点**:
1. 保存时 — IDE插件实时提示
2. git commit — pre-commit hook自动扫描 (推荐)
3. git push — CI pipeline门禁 (最后防线)

**前移原则**: 检测点越靠前，修复成本越低。pre-commit拦截=改一行代码；CI拦截=重走pipeline；生产事故=轮换密钥+评估泄露+合规处罚。

### Pre-commit Hook设计精髓

1. **只扫描staged changes**而非全仓库 — 快(<1s)+精准(只查新增)
2. **密钥扫描是阻断级，代码模式是警告级** — 渐进增强，不因误报逼用户跳过hook
3. **不依赖husky** — 直接管理.git/hooks/文件，减少依赖

### 三个关键Bug (我在审查中修复的)

1. **Hook扫描删除行**: 原版扫描完整diff包括`-`行。删除密钥是安全操作，不应阻断。修复: 加`grep -E '^\+'`只查新增行。
2. **文档中的正则教学示例触发扫描**: zhou-security-02.md中的`sk-[a-zA-Z0-9]{20,}`正则教学模式匹配了自身。修复: 字符串拆分。
3. **审计报告引用历史密钥**: zhou-v7-003-audit.md中引用了已轮换的真实密钥作为证据。修复: 用`<已轮换-XX>`占位符。

### CI Pipeline设计

GitHub Actions三层扫描:
1. 全仓库密钥扫描(比pre-commit更彻底)
2. npm audit --audit-level=high (阻断high+critical)
3. ESLint安全规则 + TypeScript类型检查 + 测试

**双重防线**: 本地hook可被--no-verify绕过→CI是不可绕过的最后防线。

## 跨模块连接

### 连接1: 安全红线 ↔ 文案温度

两条线交汇在"不能对用户说的话":
- 安全说: 不诊断、不恐吓、不绝对、不命运
- 文案说: 不烫(9-10°)、不操纵、不定义用户

**本质相同**: 都是对"边界"的定义。安全定义技术边界，文案定义情感边界。

### 连接2: 学习标准L1/L2/L3 ↔ 文案情感L1-L4

两个L体系独立但结构同构:
- 学习L1(复现)=文案L1(认知): 能跟着做/能准确描述
- 学习L2(改进)=文案L2+L3(情感+价值): 能理解为什么并优化/能解释并释然
- 学习L3(教学)=文案L4(行动): 能带别人做/能帮用户表达

### 连接3: 安全左移 ↔ 情感温度校准

安全左移=把检测提前到commit前
情感温度校准=把审查提前到发布前
**本质**: 两者都是"在低成本阶段发现问题"

### 连接4: CI门禁 ↔ 文案审查流程

安全CI有三个门: 密钥扫描→依赖审计→ESLint
文案也应该有三个门: 安全红线检查→温度越界检查→第二人审查(6°以上)
