**版本**: 1.0.0 | **任务**: V7-W4-009

---

## 设计原则

安全架构不是堆规则，是分层防守。借鉴纵深防御(Defense in Depth)思想，将安全检测按破坏力分层，每层有独立的触发条件、默认动作、以及逃生门。

```
城墙 (L1 阻断) → 护城河 (L2 警告) → 烽火台 (L3 记录)
  不可逾越          可以涉水           远见预警
```

**核心信条**:
1. 阻断必须精准——高误报率的规则放L2/L3，不放L1
2. L1规则数量≤10条——阻断规则越少越好，多了开发者会绕过(--no-verify)
3. L3是迭代引擎——记录数据积累→提炼规则→提升到L2→验证后升级到L1
4. 每一层都有逃生门——明确的override条件，避免成为开发阻塞点

---

## 三层架构总览

| 层级 | 名称 | 触发条件 | 默认动作 | 逃生门 |
|------|------|---------|---------|--------|
| **L1** | 阻断层 | 确认级威胁 | 提交失败 (exit 1) | 仓库管理员审批 + 临时白名单(≤24h) |
| **L2** | 警告层 | 疑似威胁/模糊违规 | 终端警告 + JSON标记 | 开发者确认 + 文档注释声明 |
| **L3** | 记录层 | 信息/趋势/代码气味 | JSON记录 + 趋势面板 | 无需手动处理，自动聚合 |

### 分层判定矩阵

```
发现匹配
  │
  ├─ 是否为确认级威胁(误报率<1%)？
  │    └─ YES → L1 阻断层
  │
  ├─ 是否为疑似威胁(误报率1-10%)？
  │    └─ YES → L2 警告层
  │
  └─ DEFAULT → L3 记录层
```

## L1 阻断层 — 城墙

**职责**: 拦截可导致安全事故的确认级威胁。误报必须<1%。

**规则数上限**: 10条（当前6条）

### 触发条件

| 编号 | 规则 | 严重性 | 为什么是L1 |
|------|------|--------|-----------|
| L1-01 | API Key硬编码 (`sk-`/`AKIA`/`ghp_`等) | critical | 密钥泄露=安全事故，误报率接近零 |
| L1-02 | PEM私钥 (`-----BEGIN * PRIVATE KEY-----`) | critical | 私钥不可逆，泄露需立即轮换 |
| L1-03 | .env文件提交 | critical | 配置即代码，含生产密钥 |
| L1-04 | 禁用词直写 (27项原词) | critical | 合规红线，审核直接拒绝 |
| L1-05 | 用户隐私数据硬编码 (身份证/手机号/银行卡) | critical | 违反《个人信息保护法》 |
| L1-06 | eval() 直接执行用户输入 | critical | 远程代码执行(RCE)，OWASP #3 |

### 动作
- **pre-commit**: 阻断提交 (exit 1)
- **CI pipeline**: 阻断PR合并
- **通知**: 立即通知EM(聂富贵)

### 逃生门
1. 仓库管理员(聂富贵)审批 → 在`.claude/security-whitelist.json`中添加临时白名单(≤24小时自动失效)
2. `--no-verify`跳过 → 在CI pipeline中被二次拦截(防线不依赖本地hook)
3. 白名单格式: `{"pattern":"PAT-001","file":"path/to/file","reason":"...","expires":"ISO8601","approver":"nie"}`

## L2 警告层 — 护城河

**职责**: 标记可疑行为，大概率是问题但不能100%确证。开发者可见但可自行判断。

### 触发条件

| 编号 | 规则 | 严重性 | 为什么是L2不是L1 |
|------|------|--------|----------------|
| L2-01 | 同音字禁用词变体 (329条) | high | 部分变体可能是正常文本命中（如"涨纹"=布料涨价+纹路） |
| L2-02 | 语义替换禁用词 ("真命"/"命定之人"/"一夜有钱") | high | 语义判断需要上下文，纯模式匹配有误报 |
| L2-03 | 拼音/字母变体 ("zhangwen"/"suanming") | high | 拼音可能是正常英文文本 |
| L2-04 | URL参数含token/secret | high | 可能是第三方API文档中的示例URL |
| L2-05 | HTTP Basic Auth | medium | 可能是本地开发调试 |
| L2-06 | 包管理器令牌赋值 | high | 可能是CI配置文件(ci.yml)中的secret引用 |
| L2-07 | console.log(process.env.*) | high | 开发调试习惯，不一定是泄露 |
| L2-08 | SQL字符串拼接 (非参数化) | high | 可能是静态SQL不含用户输入 |
| L2-09 | innerHTML直接赋值 | high | 可能赋值的是静态HTML模板 |

### 动作
- **pre-commit**: 输出终端警告(黄色)，不阻断提交
- **CI pipeline**: 在PR评论中自动添加警告标签
- **通知**: 日报汇总，不实时通知

### 逃生门
1. 开发者在代码所在行添加注释: `// security: known-safe — <原因>`
2. 常见safe场景: 测试文件中的假密钥、文档中的示例URL、CI配置中的secret引用
3. 每季度安全审查时评估L2→L1升级候选

## L3 记录层 — 烽火台

**职责**: 收集代码气味和趋势数据。不做即时干预，用于长期体系改进。

### 触发条件

| 编号 | 规则 | 严重性 | 为什么是L3 |
|------|------|--------|-----------|
| L3-01 | 缺少安全响应头 (X-Content-Type-Options等) | info | 缺失不影响正常运行，累积风险 |
| L3-02 | 使用已弃用的API (deprecated) | info | 版本演进中的技术债 |
| L3-03 | 依赖包过期>6个月 | info | 安全漏洞的潜在来源 |
| L3-04 | TODO/FIXME/HACK注释含安全关键词 | info | 标记了但未完成的安全修复 |
| L3-05 | 文件权限过宽 (chmod 777) | info | 开发环境习惯 |
| L3-06 | 缺少输入长度限制 | info | 潜在的DoS攻击面 |
| L3-07 | 硬编码内网地址 (localhost/127.0.0.1/192.168.) | info | 部署到生产时可能失效 |
| L3-08 | 无CSP/CSRF保护的API路由 | info | Web安全基线 |

### 动作
- **pre-commit**: 静默记录到JSON，终端不输出(保持清洁)
- **CI pipeline**: 追加到`.claude/security-trends.json`，不阻断
- **通知**: 每周安全趋势报告

### 逃生门
- 无需手动逃生，L3只记录不阻止
- 每季度安全审查时评估L3→L2升级候选

## 规则流转机制 (L3→L2→L1)

```
L3 收集180天
  │
  ├─ 分析误报率 → <5% → 候选升级到L2
  │
  └─ L2 运行90天
       │
       ├─ 分析误报率 → <1% → 候选升级到L1
       │
       └─ L1 持续监控 → 误报>1% → 降级回L2
```

| 日期 | 规则 | 原层级 | 新层级 | 理由 |
|------|------|--------|--------|------|

| 2026-05-06 | Token in URL (PAT-010) | L1 | L2 | 教学文档中的API示例URL易触发误报 |
| 2026-05-06 | 同音字替换(240条) | — | L2 | 首次部署，需要90天观察误报率 |
