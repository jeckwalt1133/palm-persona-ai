# 跨学习审查：王富贵 A/B 测试框架 — 工程视角

## 总体评价

王富贵的 A/B 测试框架在**产品方法论层面**质量很高——三组对照实验覆盖了温度理论的底层逻辑验证(H1)、边界探索(H2)、天花板验证(H3)，且实验3的安全三阶段设计体现了 PM 的伦理意识。但**从工程实现视角**，框架存在6个具体问题，其中2个可能导致实验得出错误结论。

## 审查意见

### 意见1: 用户级随机分配缺乏跨平台实现规范 `[行38-43]`

**问题**:

> `随机单位: 用户级（同一用户始终看到同一版本）`

这个需求在掌心人格局的三端(微信小程序/抖音小程序/H5)上有完全不同的实现路径:
- **微信小程序**: 用户标识依赖 `wx.login()` → `openid`，需要后端存储 `openid → version_assignment` 映射。且 `openid` 在不同小程序间不同。
- **抖音小程序**: 使用 `tt.login()` → `code` 换取 `openid`，token 机制与微信不兼容。
- **H5**: 依赖 `localStorage` + `fingerprint`，但用户清缓存或换浏览器后分配丢失。

**影响**: 如果简单用 `Math.random()` 在前端做分配，同一用户跨设备/清缓存后会看到不同版本——数据被污染。如果等后端统一方案，埋点接入(Day 5-6)的排期低估了至少3-5天的后端开发量。

**建议**:
```
定义 ExperimentUserID 接口层:
  - 微信: wx.login() → 后端换取 openid → hash(openid + experimentId) % 2 → 分组
  - 抖音: 同上，替换为 tt.login()
  - H5: localStorage 缓存 + 服务端 fingerprint fallback
统一在后端做分配决策，前端只读取已分配的分组标签。
```

### 意见2: 埋点 dwell_time 在移动端不可靠 `[行54, 行362-371]`

**问题**:

> `停留时长: 从文案曝光到用户滑动离开的时间差 | 前端曝光+离开埋点`

这个定义在移动端有三个边界情况无法处理:
1. **应用切后台**: 用户看文案时切到微信消息→计时是否暂停？如果不暂停，切后台30秒后被计为"30秒停留"——数据虚高。
2. **应用被杀**: 用户在文案区域直接关掉小程序→`copywriting_dwell` 离开事件从未发送→该次曝光的停留时长为 null，不是0。
3. **滚动回看**: 用户滑过文案后又滑回来→第二次曝光算新事件还是累计？

**影响**: 实验1中 `温5°预期停留≥12s` 这个阈值——如果虚高的切后台时间占3-5s，温5°的12s达标可能不是因为文案好，而是因为用户切出去回消息了。

**建议**:
```javascript
// 使用 Page Visibility API + 心跳机制替代简单的事件对
let dwellStart = Date.now();
let accumulatedMs = 0;
let lastActive = dwellStart;

document.addEventListener('visibilitychange', () => {
  if (document.hidden) {
    accumulatedMs += Date.now() - lastActive; // 冻结计时
  } else {
    lastActive = Date.now(); // 恢复计时
  }
});
// 每5秒发送心跳，应用被杀后服务端可根据最后心跳估算下限
```

### 意见3: 样本量与日活的矛盾 — 缺乏小样本替代方案 `[行139-149 vs 行458-462]` `[方法论问题]`

**问题**:

框架在两个地方自相矛盾:
- **行139-149**: 实验1需要160用户，建议运行3-5天(日活50+)或1-2天(日活200+)
- **行458-462**: 风险表承认"日活太低，样本量不足"的概率为**高**

如果日活50，实验1每天新增50个独立用户，3天=150——刚好到160的线。但如果日活只有20呢？实验跑8天？实验2需要250用户，日活50时跑5天只有250——但实验2的窗口是5-7天。

**这是方法论层面的问题**: 框架假设了 frequentist 统计范式(固定n, α=0.05, 1-β=0.80)，但没有考虑:
1. 小样本下 frequentist 方法的假阳性率会失控
2. 序贯分析(sequential analysis)允许边收集边检验，更适合低日活场景
3. Bayesian 方法可以用先验信息补充数据不足

**建议**:
- 增加 Bayesian 分析作为小样本补充方案: 使用弱信息先验(从冷2°的历史分享率估计)，后验分布的直接比较不依赖固定n
- 或增加序贯分析方案: 每收集30个样本做一次中期分析，用校正后的α(Lan-DeMets spending function)控制整体Type I error
- 在实验配置 JSON 中增加 `minSampleFallback: "bayesian"` 字段

### 意见4: 实验间依赖无数据版本控制 `[行451-454]` `[工程问题]`

**问题**:

> `实验2依赖于实验1的结果（如果实验1中温不优于冷，理论地基动摇，实验2需要重新设计）`

这个依赖链有脏数据风险。假设:
1. 实验1 Day 1-3 收集的数据显示"温5°优于冷2°"→实验2按原设计启动
2. 实验1 Day 4-5 新增数据反转了结论→但实验2已经在跑了，且用的是"温优于冷"的假设设计的对比(温3°vs热7°)
3. 此时实验2的数据无法独立解释——它建立在一个已失效的前提上

**建议**:
```json
// 实验配置增加版本依赖字段
{
  "experimentId": "V7-W2-005-EXP2",
  "dependsOn": {
    "experimentId": "V7-W2-005-EXP1",
    "expectedConclusion": "warm_beats_cold",
    "conclusionVersion": 1,
    "onParentInvalidation": "pause_and_review"
  }
}
```
并且实验2的分析脚本应检查 `dependsOn` 链的有效性——如果上游结论版本号变更，实验2数据标记为"待重新解释"。

### 意见5: 实验3的"烫"实验设计隐含确认性偏差 `[行236-263]` `[方法论问题]`

**问题**:

三阶段安全递增设计是优秀的伦理框架。但实验目的被定义为:

> `不证明"烫是坏的"，温度计的上限就没有数据支撑`

这句话暴露了确认性偏差——实验被设计为**验证一个已知结论**(烫是坏的)，而非**探索一个未知问题**(是否存在偏好烫文案的用户群体？)。

具体表现:
- 烫10°文案(行303-323)明确触犯了4条红线——这是一篇被**设计为注定失败**的文案
- 如果实验3得出"烫不如热"的结论，它不能证明"烫文案不好"——它只能证明"这篇被故意写坏的文案不好"
- 真正科学的问题: 在触犯0条红线的前提下，最热的文案能做到什么程度？是否存在N=1的极端案例——某个用户恰好需要烫9°的强度才能感到被理解？

**这不是否定实验3的价值**——方向验证实验(证明"这个极端是坏的")有其价值。但框架应该明确标注: 实验3是**边界验证实验**，不是**最优温度探索实验**。最优温度探索需要一个额外的实验(实验2.5): 在温3°到热7°之间做精细梯度(每个1°一个版本)，用响应曲面法(response surface methodology)找拐点。

**建议**:
- 在框架概述(§1.2)中增加 H4 假设: "存在一个最优温度区间，在该区间内共鸣评分↑但舒适度评分不↓——此区间可能在温3°-热5°之间，而非热7°"
- 将实验3的定位从"验证天花板"改为"验证天花板 + 标注方法论局限性"
- 增加实验2.5(可选): 梯级温度测试(3°/4°/5°/6°/7°五个版本)，找到共鸣/舒适度的帕累托前沿

### 意见6: 自动终止开关缺乏工程实现规范 `[行341-342, 行395-398]`

**问题**:

> `自动终止条件已编码到实验开关（冒犯率>5% → 自动关停烫组）`

"已编码到实验开关"在工程上至少需要4个组件:
1. **实时指标管道**: 每次 `copywriting_discomfort` 事件触发后，计算当前 `discomfort_count / exposure_count`
2. **阈值触发器**: 舒适率超过5%时发送终止信号
3. **终止执行器**: 收到终止信号后，将烫组流量切到热组(或关闭烫组)
4. **告警通知**: 推送通知给4人团队

当前框架只定义了 JSON config 中的占位字段(`autoKillSwitch: null`)，没有定义这些组件的实现方式。如果"编码到开关"是手动检查 dashboard——那延迟可能是几小时，期间烫文案继续曝光给新用户。

**建议**:
```json
// 实验3的autoKillSwitch应定义为
"safety": {
  "autoKillSwitch": {
    "metric": "discomfort_rate",
    "threshold": 0.05,
    "evaluationIntervalSeconds": 60,
    "minimumExposures": 10,
    "action": "redirect_to_version_A",
    "notifyChannels": ["team-dd", "team-wechat"]
  }
}
```
并且在实施路线图中增加: "Day 5.5: 自动终止开关端到端测试(模拟冒犯率>5%触发)"

### 意见7: 缺少A/A测试阶段 — 改进方案 `[新增建议]`

**问题**:

框架直接从A/B实验开始，跳过了A/A测试(两组用户看到完全相同的版本)。A/A测试是行业标准实践:
- 验证随机化机制是否正确(如果A/A测试中两组出现"显著"差异，说明随机化有问题)
- 估计指标的自然方差(为A/B的样本量计算提供真实的方差估计，而非假设值)
- 验证埋点管道端到端工作正常(两个组的数据收集逻辑是否一致)

**建议**:

在实验1之前增加一个48小时的A/A测试:
```
版本A: 冷2° 文案
版本B: 冷2° 文案 (与A完全相同，但标记为不同version_id)
预期: 两组在所有指标上无显著差异
通过标准: 所有指标的p-value > 0.10 (即不能拒绝"两组相同"的零假设)
```
- 样本量: 40-60用户(不需要达到A/B的样本量，A/A只需要检测随机化偏差)
- 时间: 1-2天
- 成功后version_id将被验证为不可区分，后续实验的差异可归因于文案本身
