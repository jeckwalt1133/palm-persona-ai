## 背景

当前AI师生研究院面临一个根本性瓶颈：

**每次Claude Code会话上下文压缩后，团队失去对话历史中的关键信息——决策过程、任务进度、中间发现、被否决的方案。** bootstrap.md是手动快照，依赖主会话人工维护。这不是可持续的。

团队4个AI实例（主会话聂富贵+独立tmux马富贵+将来启动的王富贵+周富贵）需要一套**自动化的永久记忆系统**，使得：
1. 上下文压缩后，团队可以自动恢复工作状态，不丢失关键信息
2. 每个成员的各领域记忆不互相覆盖
3. 任务在成员之间自动流转，避免闲置

这本质上是PROTOCOL.md三层记忆中Layer 2和Layer 3从来没有真正实现的问题。

### 研究目标

设计一套**自动化上下文管理+团队记忆系统**，目标状态：

```
┌─────────────────────────────────────────────────────────────┐
│  目标状态                                                    │
│                                                              │
│  会话中 → PreCompact hook 自动快照                           │
│       ├─ 当前任务进度 + 阻塞项                              │
│       ├─ 本次会话的关键决策 (decision log)                  │
│       ├─ 本次会话的关键发现 (discovery log)                 │
│       └─ 下次会话的启动指令 (resume prompt)                 │
│                                                              │
│  压缩后 → SessionStart hook 自动恢复                         │
│       ├─ 读取上次快照 → 知道做到哪了                        │
│       ├─ 读取团队状态 → 知道其他人在做什么                  │
│       ├─ 读取任务队列 → 知道下一个任务是什么                │
│       └─ 自动继续被打断的工作                               │
│                                                              │
│  团队间 → 共享记忆同步                                       │
│       ├─ 主会话快照 → 学生会话可读取                       │
│       ├─ 学生产出 → 主会话自动感知                         │
│       └─ 客座审查 → 结构化存入记忆                          │
└─────────────────────────────────────────────────────────────┘
```

### 具体研究问题

1. **压缩前快照（PreCompact）应该保存什么？**
   - 哪些信息是"丢了就无法恢复"的？
   - 哪些信息是"可以从文件中重新读取的"（不需要保存）？
   - 快照格式应该是什么？（结构化JSON？Markdown摘要？）

2. **压缩后恢复（SessionStart）应该怎么自动恢复？**
   - 如何判断"上次做到哪了"？
   - 如何自动读取正确的文件？
   - 如何避免把已经完成的工作再做一遍？

3. **团队记忆如何分层存储？**
   - 哪些信息是全局共享的（bootstrap）？
   - 哪些信息是各角色私有的（domain memory）？
   - 如何避免记忆冲突（两人对同一件事的记录不一致）？

4. **任务如何自动流转而不依赖人工？**
   - 老师布置任务 → 学生完成 → 客座审查 → 老师终审
   - 这个流程中，哪些环节可以自动化？
   - 如何让学生"主动拉取"任务而不是"被动等待注入"？

5. **如何做到"永久可持续"？**
   - 记忆文件的增长管理（不会无限膨胀）
   - 过期记忆的清理策略
   - 关键信息永不丢失的保证机制

### 交付物

一个文件：`research/memory-automation-design.md`

包含：
1. **问题分析**：当前系统的具体断点（哪些环节丢了信息）
2. **设计方案**：完整的自动化记忆系统架构
   - 快照格式定义
   - 恢复流程设计
   - 分层存储方案
   - 任务流转协议
3. **实现路径**：分3个Phase的实现计划
   - Phase 1: 最小可行（当前会话能用的）
   - Phase 2: 团队协作（多人同步的）
   - Phase 3: 完全自动化（无需人工的）
4. **文件清单**：需要创建/修改哪些文件

### 约束条件

- 记忆文件只存D盘（`/mnt/d/Claude/Workspace/palm-persona-ai/memory/`）
- 不能依赖外部数据库（SQLite FTS可以，但不要MySQL/Redis）
- 快照必须能在3秒内生成（不能阻塞PreCompact hook）
- 恢复必须能在30秒内完成（不能阻塞SessionStart hook）
