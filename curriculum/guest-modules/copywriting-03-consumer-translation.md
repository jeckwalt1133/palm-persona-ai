## 安全红线 (Safety Notes)

技术翻译的安全边界与文案创作不同，但同样重要：

1. **不失真** — 翻译不能改变技术事实。把"Mock数据"翻译成"真实数据"是欺骗。
2. **不模糊** — 翻译不能把有确定含义的术语变成一个模糊词。把"降级"翻译成"切换"丢失了"从高到低"的方向信息。
3. **不自嗨** — 翻译目标不是炫文案技巧，是让目标读者理解。如果你翻译后的版本比原文更难懂，翻译失败。
4. **不甩锅** — "这是技术细节，用户不需要懂"不是翻译策略。用户不需要全懂，但需要懂到能做出判断的程度。

## 第一部分：判断什么需要翻译

### 1.1 三级读者模型

不是所有术语都需要翻译。取决于这段文字的**目标读者**是谁。

```
Level A — 工程读者（马富贵/周富贵）
  → 术语保留。hash/Pipeline/MCP都是日常词汇。
  → 你的工作：零。不需要翻译。

Level B — 产品/设计读者（王富贵/产品经理/设计师）
  → 术语翻译。需要理解"这个技术做什么"，不需要理解"怎么实现"。
  → 你的工作：把实现细节转为功能描述。

Level C — 终端用户（小程序使用者）
  → 术语消失。用户不需要知道任何技术概念。看到的是结果。
  → 你的工作：把功能描述转为用户价值。
```

### 1.2 必须翻译 vs 可以保留

| 保留（工程读者可理解） | 必须翻译（跨领域读者需要） |
|----------------------|--------------------------|
| API / JSON / CSS — 行业通用缩写 | hash / O-W / Worker — 领域特定缩写 |
| Canvas — 前端通用术语 | 海马体重放 — 神经科学隐喻 |
| MCP — 行业标准协议名 | 5Worker流水线 — 实现细节编号 |
| 降级链 — 含义可推断 | Mock(基于hash随机) — 两重黑话叠加 |

**判断标准**：找一个不是工程师的团队成员，问他这个词什么意思。如果他说不出来，就需要翻译。

## 第三部分：工程→产品翻译的5条原则

### 原则1: 功能先行 — 「说它做什么，不说它怎么做」

工程语言描述**实现**，产品语言描述**功能**。

```
❌ 工程: "通过Sharp库提取掌纹像素特征并输入5Worker Pipeline"
✅ 产品: "系统读取你的掌纹，分析其中的特征模式"

❌ 工程: "MCP Server暴露check_compliance标准接口"
✅ 产品: "内容安全检查功能对团队内部工具开放调用"
```

**检查方法**: 删除所有技术名词（库名/框架名/协议名/模式名）后，句子是否仍然完整传达了功能？不完整→补功能描述。

---

### 原则2: 读者视角 — 「用读者的认知框架，不用你的」

工程师说"降级链"，在工程框架里这个词是精确的。但产品经理听到"降级"想到的是"服务降级/体验下降"——一个负面的东西。

```
❌ 工程认知: 降级=容错机制，是好事
⚠️  产品认知: 降级=降低了标准，是坏事
✅ 翻译: "智能切换（主模型不可用时自动换备用）"——既准确又不触发负面联想
```

**常见认知错位**:

| 工程词 | 工程师想到的 | 外行想到的 | 应翻译为 |
|--------|-----------|-----------|---------|
| 降级 | 容错保障 | 质量下降 | 智能切换/自动备选 |
| 门禁 | 安全检查点 | 被拦住不让进 | 内容审核/安全过滤 |
| 流水线 | 数据处理步骤 | 工厂/机械 | 处理流程/生成步骤 |
| Mock | 测试替身 | 嘲笑？假？ | 模拟/演示版 |
| Worker | 计算单元 | 打工人？ | （删除，用"步骤"代替） |

---

### 原则3: 承诺对齐 — 「命名不承诺你还没做到的事」

这是 V7-W3-002 审查中最核心的发现。"语义知识图谱"的命名承诺了 NLP 级别的语义理解——但实际实现是关键词匹配。**名字承诺的东西，实现没有给。**

```
命名承诺的    实现交付的    信任差
─────────────────────────────────────
语义搜索      关键词匹配    负（过度承诺）
AI引擎        if/else规则   负（过度承诺）
智能推荐      随机排序      负（过度承诺）
知识索引      关键词检索    零（准确对齐）
自动化规则    Cron脚本      零（准确对齐）
```

**规则**: 功能到哪，命名到哪。实现升级了，命名跟着升级。顺序不能反。

**实际案例**（来自本项目的知识图谱）:
```
当前命名: 语义知识图谱 (semantic-knowledge-graph)
当前实现: JSON文件 + 关键词grep + --domain筛选
命名承诺: 语义理解（同义词/上下文/模糊匹配）
实现交付: 字符串精确匹配
→ 信任差: 用户搜"记忆"找不到"memory"节点——语义？不语义。

建议改名: 项目知识索引
建议时机: 现在
升级条件: 接入向量检索后 → 可升级命名为"语义知识图谱"
```

---

### 原则4: 渐层降维 — 「代码术语→工程术语→产品术语→用户语言，逐级翻译」

不要一步从代码跳到用户语言——中间会丢失太多信息。每一步降一层。

```
Layer 0: 代码        "Sharp库 extract() → 5Worker Pipeline"
                        ↓ 降维
Layer 1: 工程概念    "掌纹特征提取 → 五步处理流程"
                        ↓ 降维
Layer 2: 产品概念    "读取掌纹特征 → 自动生成分析报告"
                        ↓ 降维
Layer 3: 用户语言    "扫一扫，立刻了解你的性格特点"
```

**每一层去掉什么、保留什么**:

| 降维 | 去掉 | 保留 | 示例 |
|------|------|------|------|
| L0→L1 | 库名/函数名/具体实现 | 模块名/步骤数/数据流向 | Sharp→"特征提取" |
| L1→L2 | 技术架构/模块划分 | 功能/输入输出/效果 | "五步处理"→"自动生成" |
| L2→L3 | 所有技术概念 | 用户能获得什么 | "分析报告"→"了解性格" |

---

### 原则5: 可验证 — 「翻译后让一个外行读，他能复述出正确的理解」

翻译的质量标准不是"我觉得翻得好"，是"目标读者看完后理解正确"。

**验证方法**（两步）:

```
步骤A — 找目标读者（或另一个AI实例扮演目标读者）
步骤B — 让他读完翻译后回答三个问题：
  Q1: 这段话说的是什么功能？（功能理解）
  Q2: 这个功能对我有什么用？（价值理解）
  Q3: 你觉得这个功能有什么做不到的地方？（边界理解——如果答"什么都能做"，翻译夸大了）
```

**通过标准**: Q1和Q2正确 + Q3不产生错误预期 = 翻译合格。

## 常见问题

**Q: 有些术语是行业标准（如MCP），翻译了别人反而不认识怎么办？**
保留原名+括号注翻译。例如"MCP协议（行业标准的模型上下文协议）"。第一次出现时注释，后续直接用原名。这既尊重了行业惯例，又降低了新人的理解门槛。

**Q: 产品层翻译和用户层翻译的边界在哪里？**
产品层：团队成员看——需要知道"是什么"和"为什么"。用户层：用户看——只需要知道"对我有什么用"。产品层可以保留"AI模型""安全检查"等概念词，用户层这些词都要转化为"智能""安全"等感知词。

**Q: 翻译后发现原文的意思我不确定，怎么办？**
这说明你还没理解那个技术概念。**翻译的前提是理解。** 不确定就回去问工程师。强行翻译一个没理解的概念，翻出来的一定是错的。

**Q: 5条原则都要用吗？**
不是每条翻译都需要5条全用。原则1-3是每次必查的（功能先行/读者视角/承诺对齐），原则4是多层翻译时用的，原则5是翻译完成后验证用的。
