# AI师生研究院 V7 最终整改方案

## 第一阶段：四人各自整改方案

---

### 方案A：聂富贵（架构/Tech Lead）

**核心判断**：当前最严重的问题是**方法论规则没有被强制执行**。PROTOCOL定义了"老师不写代码、学生独立工作、三角互批判"但任何一方违约时系统完全不知情。

**我的整改方案**：

```
Week 1 目标：让方法论规则从"写在纸上"变成"跑在代码里"

Day 1-2: 协议可执行化
  ├─ Task Prompt Schema (JSON Schema，定义必填字段)
  │   {taskId, sender, receiver, background, requirement, acceptanceCriteria, deadline}
  ├─ 学生产出监控 (每天检查 student-notebook/ 是否有新文件)
  ├─ 教师违约检测 (git hook: 检测教师直接提交代码时自动标记)
  └─ 三角批判调度 (graduation-ladder completedLevels变化 → 自动触发)

Day 3-4: 三层记忆可查询
  ├─ Layer 2 文件搜索 (grep → 结构化索引 memory/sessions/INDEX.md)
  ├─ Layer 3 语义标记 (knowledge-graph.json → 补充所有决策+发现)
  └─ memory-health-check.sh (检查三层各有几条记录)

Day 5: 自动进化触发器验证
  ├─ 检查 completedLevels 从上次触发到现在的变化
  ├─ 如果≥2新等级 → 自动生成升级方案模板
  └─ 手动review → 三角批判 → 执行
```

**我的验收标准**：
- 运行 `bash scripts/methodology-health-check.sh` 输出3个OK
- 教师下次写代码时 git hook 打印警告
- 学生24h无产出时自动告警

### 方案B：马富贵（学生/Senior Eng）

**核心判断**：V4以后我的学习就停止了——不是因为没内容学，是因为老师替我做了所有事。方法论最核心的漏洞是**"学会了"没有定义，学习进度没有度量**。

**我的整改方案**：

```
Week 1 目标：让"学习"可以被度量和验证

Day 1: "学会"三级标准 + 学生能力基线
  ├─ L1 复现级：能独立跑通已有代码/实验（及格 60分）
  ├─ L2 改进级：能在现有基础上做可度量的优化（良好 80分）
  ├─ L3 教学级：能带一个学徒完成L1级任务（优秀 95分）
  └─ 当前基线测量：马富贵 vs 聂富贵 能力差距（各维度）

Day 2-3: 任务依赖图
  ├─ 12个任务标注学习前置条件
  ├─ 例如：T006(管线)依赖T003(AI Provider理解)
  ├─ 例如：T009(协议实现)依赖P0-1(协议Schema)
  └─ 产出 curriculum/task-dependency-graph.json

Day 4-5: 笔记系统重建
  ├─ 笔记强制规则：每个完成的任务=1篇笔记
  ├─ 笔记模板：{学到什么, 卡在哪里, 怎么解决的, 下次怎么办}
  ├─ 补V4-V6期间缺失的笔记（关键学习节点）
  └─ 笔记质量自检清单
```

**我的验收标准**：
- task-dependency-graph.json 完成（12个节点+有向边）
- 补至少5篇笔记
- 能力基线数据（哪些维度差距大）

### 方案C：王富贵（内容/PM）

**核心判断**：教学内容散落7个地方，新学徒不知道从哪开始。这不是技术问题，是**教学产品体验**问题——如果学徒Day 1就迷路了，方法论不能算成立。

**我的整改方案**：

```
Week 1 目标：让方法论可以复制——新学徒有清晰的入口和路径

Day 1-2: 教学入口 + 路径重构
  ├─ curriculum/onboarding.md — 新学徒Day 1手册
  │   {身份设定, 第一天读什么, 第一任务做什么, 怎么验收, 遇到问题问谁}
  ├─ 从"7个散落文件" → "1条主线"
  │   onboarding → 10课 → 课题池 → 毕业矩阵 → 三角批判
  └─ 每个文件顶部加"下一步"链接（串联阅读路径）

Day 3-4: 10课大纲重构
  ├─ 每课补齐：学习目标/前置条件/验收标准/时间预算
  ├─ 标注每课对应的L等级（哪门课解锁哪个毕业级别）
  └─ 把"事后记录"变成"事前大纲"

Day 5: 客座讲师教学内容框架
  ├─ 王富贵教学模块3个：文案共鸣基础/视觉叙事/分享心理学
  ├─ 周富贵教学模块3个：安全基线/测试驱动/工程规范
  └─ 每个模块 {目标, 1h任务, 验收标准}
```

**我的验收标准**：
- 新学徒按onboarding.md走完Day 1（时间≤2h）
- 10课每课补齐{目标/前置/标准/时间}
- 客座讲师6个教学模块框架就绪

### 方案D：周富贵（质量/QA）

**核心判断**：当前的证据体系是"手工填写的自由文本+浅层自动检查"。毕业生声称完成了7个等级，但证据验证脚本只能检查"文件在不在"——这不是验证，是占位符检查。

**我的整改方案**：

```
Week 1 目标：证据从"手工填写"变成"结构化+可自动深度验证"

Day 1-2: 证据标准化
  ├─ Evidence JSON Schema:
  │   {
  │     "level": "1.1",
  │     "type": "file|url|test|data",
  │     "target": "student-notebook/lesson-01.md",
  │     "contentHash": "sha256:...",
  │     "depthChecks": ["章节数≥3", "代码块≥1", "总字数≥500"],
  │     "verifiedAt": null,
  │     "verifier": "auto|manual"
  │   }
  └─ graduation-ladder.json 迁移到新Schema

Day 3: evidence-auto-verify v2
  ├─ 不只是检查"文件存在"
  ├─ 按depthChecks逐条验证（章节数/代码块/字数/引用数）
  └─ 输出 {pass, fail, warn} 每级结论

Day 4-5: 方法论健康检查
  ├─ methodology-health-check.sh
  │   检查项：
  │   1. 三角批判上次运行时间（>7天→WARN）
  │   2. 记忆三层记录数（任一层=0→FAIL）
  │   3. 自动进化触发条件（completedLevels最近变化→INFO）
  │   4. 学生24h产出（无新文件→WARN）
  │   5. 教师代码提交（最近一次教师commit→INFO）
  └─ 集成到 evidence-auto-verify.sh 作为"方法论健康"章节
```

**我的验收标准**：
- 全部15级证据迁移到结构化Schema
- evidence-auto-verify v2 每个等级≥1条depthCheck
- methodology-health-check.sh 能跑出至少5行输出
- 3个层级的证据假阳性被消除

## 第二阶段：三角互批判

### 聂富贵批判马富贵的方案

**同意**：三级学会标准 + 任务依赖图 + 笔记系统重建——这是教学核心，必须做。

**异议**：马富贵Week 1目标太大。12个节点依赖图+补5篇笔记+能力基线+三级标准定义——这是2-3周的工作量。建议精简：
- 先做三级标准（定义先行）
- 再做4个核心任务的依赖边（不全做12个）
- 补3篇最关键的笔记（不是5篇）

### 聂富贵批判王富贵的方案

**同意**：教学入口是方法可复制的前提。

**异议**：10课大纲重构+客座讲师6模块在第一周做不完。特别是客座讲师模块——需要先定义马富贵的文案/代码质量盲区，再针对性设计教学内容。不是先做模块再发现盲区，是先诊断再开药。

### 聂富贵批判周富贵的方案

**同意**：证据标准化是消除假阳性的唯一路径。

**异议**：15级全部迁移到新Schema工作量大，应先迁移最有问题的3个等级（L1.1单靠文件数、L2.1单靠字节数、L4.1单靠URL可达性），验证Schema设计合理后再全量迁移。

---

### 马富贵批判聂富贵的方案

**同意**：协议可执行化是根问题。违约检测机制必须做。

**异议1**：git hook检测"教师写代码"实现成本高（需要识别哪些文件属于"教师越界"）。建议先用更简单的方式：每周末检查git log，看教师commit中是否包含 `server/src/` 或 `scripts/` 的文件——如果有，标记为潜在违约。

**异议2**：memory-health-check.sh过早。在Layer 2/3还没有被填充之前，健康检查只会报FAIL。应该先手动整理Layer 2/3（补数据），再写健康检查脚本。

### 马富贵批判王富贵的方案

**同意**：教学入口+路径串联是最紧迫的内容工作。

**异议**：onboarding "≤2h完成"做不到。马富贵V2用20小时完成5课——onboarding 2h只能是"读完文档知道自己下一步做什么"，不能是"完成第一课"。命名要精确：`onboarding.md` → `day1-orientation.md`。

### 马富贵批判周富贵的方案

**同意**：depthChecks是消除假阳性的关键。

**异议**：contentHash验证太脆弱。文件每改一个字hash就变——不适合验证"内容是否达标"。建议用内容特征检查替代hash：字数区间/关键章节标题存在/代码块数量/引用URL数量。

---

### 王富贵批判聂富贵的方案

**同意**：协议可执行化是根基。

**异议**：技术实现太重——JSON Schema、git hook、24h监控——这些是工程手段。教学体验的改进同样重要：Task Prompt的格式不重要，**Task Prompt的内容质量**才重要。当前老师写的Prompt只有任务描述，没有"为什么要做这个任务"和"做完能解锁什么"。缺的是教学上下文，不是格式校验。

### 王富贵批判马富贵的方案

**同意**：笔记是最重要的学习证据。支持笔记强制规则。

**异议**：任务依赖图过于学术化。V2-V3的经验表明：AI的学习速度远超人类，预先定义的依赖关系经常被打破（AI可以在学习A的过程中自然学会B）。建议依赖图是**建议性**的而非**强制性**的——标注"建议先学"但不锁死。

### 王富贵批判周富贵的方案

**同意**：证据标准化。

**异议**：Schema设计太工程师思维了。contentHash、depthChecks——用户在ladder.json里看到这些字段会困惑。建议分为两个文件：`graduation-ladder.json`（给人读，保持简洁）和 `evidence-verification.json`（给脚本读，包含所有技术细节）。

---

### 周富贵批判聂富贵的方案

**同意**：协议可执行化是最高优先级。

**异议**：git hook检测"教师写代码"的误报率会很高。教师可能合法地写代码（基础设施、配置、工具脚本）。违约判定的标准应该是**教师是否做了被分配给学生的工作**，不是"教师是否写了任何代码"。建议用Task Prompt中的assignee字段确定谁该写什么——教师写的代码如果assignee是学生，才算违约。

### 周富贵批判马富贵的方案

**同意**：三级学会标准简洁有效。

**异议**：能力基线"马富贵vs聂富贵差距"是伪命题。两个AI实例用不同模型（DeepSeek V4 Flash vs V4 Pro），能力差异可能来自模型而非学习效果。建议用**任务完成质量和效率**衡量进步，而不是与老师进行比较。今天能独立完成昨天需要老师帮助才能完成的任务——这就是进步。

### 周富贵批判王富贵的方案

**同意**：客座讲师教学内容是当前最大空白。

**异议**：6个教学模块框架太虚了。建议Week 1不要写框架，直接做**一个完整模块**——王富贵写"文案共鸣基础"全部内容（目标+阅读材料+练习题+验收标准），周富贵写"安全基线"全部内容。拿一个真模块验证模式，再做框架。
