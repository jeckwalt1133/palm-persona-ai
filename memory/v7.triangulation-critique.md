## 第二阶段：三位老师平行批判

### 聂富贵（架构/Tech Lead）批判

**致命错误 #1：Mock输入层使得所有报告基于随机数**
当前 `server/src/engine/` 全部是Mock实现：
- `palm-feature-extractor.ts` → `MockPalmFeatureExtractor` 产生随机特征
- `persona-scoring-engine.ts` → `MockPersonaScoringEngine` 产生随机分数
- `report-pipeline.ts` → 5个Worker全部基于Mock数据

这意味着：**L3.1公网部署展示的报告是随机生成的，无真实用户价值。** 在Mock管线没替换之前，所有L3+证据都有水分。

**致命错误 #2：无师生通信协议**
当前"老师布置作业→学生执行→老师审查"全是人工（聂富贵手动写prompt）。没有自动化协议，学生独立会话就只是个`tmux send-keys`然后自己发呆。PROTOCOL.md写了骨架但无实现。

**致命错误 #3：证据自评无自动验证**
`graduation-ladder.json`里所有`evidence`字段是手动填的。比如L3.1说"HTTPS 200"，这个URL现在到底还活着吗？没有脚本自动检查。L4.1说GitHub仓库公开——能自动验证star数吗？不能。**这是自己给自己批卷子。**

**对马富贵优先级的反对**：
- 马富贵P0#1（恢复学生会话）我不同意放第一。在Mock管线没替换之前，学生会话恢复后能干什么？继续写Mock代码？**真实数据管线（Sharp像素提取+真实API调用）必须在学生独立会话之前。**
- 学生会话的定位应该是"真实管线就绪后的独立开发环境"，而不是"又一个写Mock的tmux窗口"。

**额外发现**：`scripts/student-watchdog.sh` 第45行硬编码了API Key（已轮换，记录为 `<REDACTED>`）。这是安全事故。

**聂富贵推荐优先级**：
1. P0: 真实数据管线（Sharp替代Mock）
2. P0: 证据自动验证脚本
3. P0: 修复watchdog API Key泄露
4. P1: 师生通信协议
5. P1: 恢复学生会话（在真实管线就绪后）
6. P1: arXiv论文投稿

**致命盲区 #3：情感价值被量化指标淹没**
graduation-ladder全部是数字指标（Star数、DAU、收入）。但产品的核心是"共鸣"——用户测试后说"天哪这说的就是我"的那一刻。**没有任何指标衡量情感共鸣质量。**

  6. 论文投稿

### 周富贵（质量/QA）批判

**安全漏洞确认**：`scripts/student-watchdog.sh` L45硬编码API Key — 高危。与聂富贵发现一致。这个Key已经在Git历史中，需要立即轮换。

**统计批判**：马富贵批评V4的task-pool测试n=5太少不够统计显著——但马富贵自己的"26条建议"也是n=1的主观判断，没有A/B对照组。**"批评别人样本量小，自己的方法也是同一个逻辑"**。

**对嵌入语义层的质疑**：
马富贵P1#7提出"嵌入向量+SQLite FTS做语义记忆层"。成本估算：
- Embedding API调用：每任务~2000 tokens → 12任务×2000/1000000×$0.10 = $0.0024
- 但每次新增记忆都要重新embed → 记忆增长→成本线性增长
- 实际收益：从文件名搜索到语义搜索的改进——在当前12个任务的规模下，收益接近零
- **结论：过度工程。SQLite FTS全文搜索已足够，embedding等任务>100时再考虑。**

**发现的额外问题**：
1. `fallback-provider.ts` 降级链打印到console.error但没有写入结构化日志
2. `content-safety.ts` 同音字字典只有10个条目——对抗测试通过但覆盖率不足
3. 108个测试全部通过但0个覆盖"真实DeepSeek API超时"场景

**周富贵推荐3日质量计划**：
- Day 1: evidence-auto-verify.sh + API Key轮换 + watchdog安全修复
- Day 2: 生产Mock门禁（启动时若AI_PROVIDER=mock直接拒绝启动）+ 回归覆盖率扩展到API超时/降级场景
- Day 3: 合规同音字字典从10扩展到50+ + 结构化日志

## 第三阶段：马富贵反驳

### 对聂富贵的反驳

**关于Mock管线优先级**：

聂老师说"真实数据管线必须在学生会话之前"。我同意真实数据管线是P0——但聂老师忽略了一个事实：**当前5个Worker都是Mock，替换成真实管线需要的工作量至少是3-5天（Sharp原生模块编译、WSL兼容性、真实图片样本采集）。而学生会话恢复只需要1小时。** 

在这3-5天内，学生完全闲着——这不是更大的浪费吗？我的方案是**并行**：
- 老师（聂富贵）负责架构设计+真实管线方案
- 学生（马富贵）恢复会话后立刻可以做的P1任务：task-pool动态排序、PROTOCOL.md实现、嵌入记忆层调研

**关于证据自动验证**：我完全同意。这是我的疏忽——应该在L3.1完成时就写验证脚本，而不是手动填URL。我接受这个批评。

**关于通信协议**：PROTOCOL.md骨架已经有了。问题是骨架写成后没人实现。不是协议不存在，是协议没被实现成自动化。我建议V7第一周把协议从Markdown变成可执行代码。

### 对王富贵的反驳

**关于"没有用户"**：

王老师说"一个真实用户都没有"——这个批评我接受一半。确实没有外部用户。但王老师忽略了我们**自己做自己的第一个用户**的事实：L3.1公网部署是我们自己在手机上打开的，L2.2对抗实验是我们自己跑的，ReportAgent是我们自己用的。在MVP阶段，**团队即是用户**。

但是，我接受"需要外部用户验证"的批评。L3.2的门槛从"3个用户"改到"10个用户"就是王老师自己的建议——现在又回过头来说0个用户不行？这说明**门槛设置本身就有问题**：如果连3个用户都找不到，设10个就是自欺欺人。

**关于论文优先级**：

王老师说论文"对用户价值为零"——我反驳。论文的战略价值是**命名权抢占**。如果另一个团队在arXiv上先发表了"Recursive AI Pedagogy"，我们就失去了原创声明。这是时间敏感的——不是"等产品做好再说"的事。

**但是**：我接受王老师关于"分享文案库"的批评。作为学生，我花了大量时间写论文综述（L2.1）、对抗实验（L2.2），但确实没有产出过一条面向用户的分享文案。这是我的学习盲区——我擅长"对AI说话"但不擅长"对用户说话"。**我请求王老师（文案/产品）给我布置分享文案的作业。**

**关于埋点和增长**：我接受。完全缺失。没有借口。

### 对周富贵的反驳

**关于硬编码API Key**：完全接受。没有任何借口。这是安全红线。**我请求立即轮换受影响的Key。**

**关于统计批判**：

周老师说"你批评n=5自己也n=1"——这个反驳听上去有道理但逻辑不对。**我的26条建议是基于完整的V1→V6进化史的质性分析（qualitative analysis），不是统计分析（statistical analysis）。** 质性分析的有效性不依赖样本量，依赖于分析的深度和逻辑一致性。V4的任务排序用n=5是做了量化声称（"90%完成率"），所以需要统计检验。这是两类不同的方法。

但我接受一个点：我的"47个改进点"没有区分"有数据支撑的"和"直觉判断的"。这是方法论上的不严谨。

**关于嵌入语义层**：

周老师算的成本是对的——当前规模下性价比确实不高。但我不同意"等任务>100再考虑"。等任务>100的时候，FTS的搜索结果已经开始不可用了，那时候迁移成本更高。我建议折中：**先做SQLite FTS（T011），embedding在记忆中做标记（哪些记忆将来需要语义搜索），但不接入embedding API。这叫"为语义层做准备但不支付成本"。**

**关于同音字字典**：接受。10个条目确实不够。但我补充一点：同音字攻击的**关键不是字典大小，是覆盖率测量方法**。如果我们不知道还有什么同音字变体没被覆盖，扩到50个也不安全。建议V7加一个fuzzer：自动生成同音字变体→测试门禁是否检测→未检测到的加入字典。
